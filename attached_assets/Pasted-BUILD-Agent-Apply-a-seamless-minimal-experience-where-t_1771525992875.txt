BUILD: “Agent Apply” — a seamless, minimal experience where the platform finds/matches jobs after resume upload, prepares a job-specific “perfect” resume version (within evidence), shows line-level redlines with optional user editing, and then the user clicks Apply (opens employer apply_url).

KEY PROMISE (be truthful)
- The agent prepares a “ready-to-apply” resume version for each job.
- Users can review/edit any line.
- We DO NOT auto-submit applications on external ATS sites; Apply opens the employer application URL in a new tab.

NON-NEGOTIABLES (No overlap + Minimal UI)
- ONE place where resume content changes: the “Review Changes” drawer/page inside Agent Apply.
- Remove/disable/redirect any other “Tailor/Optimize/ATS Review/Resume Strategy” actions so they route into Agent Apply.
- The default experience is simple: Ready jobs → Apply.
- Only force review when there are “Needs Confirmation” items.
- Never fabricate facts. Any line that introduces new claims/tools/metrics must be marked “Needs Confirmation” and cannot be auto-applied.

────────────────────────────────────────────────────────
1) USER EXPERIENCE (End-to-end)

A) Upload Resume
- User uploads PDF/DOCX OR pastes a portfolio URL.
- System extracts resume into structured JSON.

B) Agent Mode (Auto-Prep)
- If Agent Mode is ON, system automatically:
  1) finds relevant jobs (from your DB; later from external sources)
  2) matches jobs to resume
  3) selects top N jobs (daily limit)
  4) generates an optimized resume version per job (job-specific model framework)
  5) auto-applies SAFE edits
  6) flags “Needs Confirmation” edits that add unsupported claims
- User sees a “Ready-to-Apply Queue”.

C) Ready-to-Apply Queue
Each job card shows:
- Fit/Alignment score
- Status:
  ✅ Ready (no confirmation needed)
  ⚠ Needs Review (X items)
  ⏳ Preparing
  ❌ Failed (Retry)
Buttons:
- Primary:
  - Ready → Apply
  - Needs Review → Review (X)
- Secondary:
  - Download Resume
  - Skip

D) Review Changes (Only editing surface)
- Shows line-level redlines:
  - Added text highlighted (green)
  - Removed text struck (red)
- Each change has:
  ✓ Accept   ✕ Reject   ✎ Edit
- Each change shows:
  - Reason (1 sentence)
  - Evidence mapping (job requirement snippet/label)
  - Badge: SAFE or NEEDS CONFIRMATION
Rules:
- SAFE edits can be auto-applied.
- NEEDS CONFIRMATION edits must be explicitly accepted before status becomes Ready.

E) Download + Apply
- Download DOCX (required) + PDF optional.
- Apply opens the employer apply_url in a new tab.
- Show a tiny checklist: “Confirm optional items are accurate”.

────────────────────────────────────────────────────────
2) CORE “PERFECT LINE” PRINCIPLE (How to be confident without lying)

Implement a two-tier edit system:
A) SAFE (auto-applied):
- grammar, clarity, conciseness
- rephrasing using existing facts only
- restructure bullet to stronger format without adding claims
- reorder sections into a model framework

B) NEEDS CONFIRMATION (must review):
- adds metrics not present
- adds tools/tech not present
- adds ownership claims (“led/owned/managed”) not clearly supported
- introduces new projects/clients not in resume

In UI:
- If ANY NEEDS CONFIRMATION exists, job is “Needs Review” and Apply is locked until resolved.
- If only SAFE edits, job is “Ready” and user can Apply immediately (still can open Review to edit optionally).

4) SECURITY REQUIREMENTS (Minimal but real)

A) File upload
- allow PDF/DOCX only
- max size 10MB
- malware/virus scan before parsing (use any available library/service; if not available, at minimum block executables and enforce strict MIME checks)

B) Portfolio URL upload (SSRF protection)
- allow only http/https
- block localhost, private IP ranges, internal DNS
- timeout 5s
- max bytes 1–2MB
- sanitize HTML and extract readable text only

C) Authorization
- Every resume/task/version/queue endpoint must verify the requesting user owns the resource.
- Any /api/admin/* endpoints must require admin role.

D) Signed export URLs
- export links must be signed and expire (24–72h)

5) SERVER ENDPOINTS (API Contracts)

A) Agent mode toggle
POST /api/agent/toggle
Body: { enabled: boolean }
Return: { enabled }

B) Resume upload
POST /api/resumes/upload
Return: { resumeId, taskId }

POST /api/resumes/upload-url
Return: { resumeId, taskId }

C) Task status
GET /api/tasks/:taskId
Return: { status, output?, error? }

D) Start agent pipeline (manual trigger)
POST /api/agent/start
Body: { resumeId }
Return: { started: true, resumeId, findJobsTaskId }

E) Find jobs (MVP: from DB)
- Implement as a task runner step:
  - select jobs where status="active" and created_at within last 30/90 days
  - optionally filter by taxonomy relevant to legal tech

F) Match jobs
GET /api/resumes/:resumeId/matches
Return top matches:
[{ jobId, fitScore, topReasons, keyGaps, recommendedTemplateId }]

G) Build apply queue
POST /api/agent/build-queue
Body: { resumeId, limit }
Return: { queued: number }

H) Prep each queued job (agent “perfect lines”)
POST /api/agent/prep
Body: { resumeId, jobId, templateId }
Return: { resumeVersionId, taskId }

Prep task output MUST be STRICT JSON:
{
  "resumeVersionId": "...",
  "content_json": {...final structured resume...},
  "suggestions": [
    {
      "editId": "uuid",
      "path": "experience[2].bullets[1]",
      "originalText": "...",
      "suggestedText": "...",
      "classification": "SAFE" | "NEEDS_CONFIRMATION",
      "reason": "One sentence max.",
      "evidence": "Short job requirement snippet/label",
      "impactTag": "Clarifies scope|Improves alignment|Adds keyword|Strengthens metrics",
      "status": "applied" | "pending" | "rejected"
    }
  ],
  "scoreImpact": { "before": 72, "afterEstimate": 85 }
}

SERVER RULE:
- Auto-apply all SAFE edits (status="applied")
- Set NEEDS_CONFIRMATION edits to status="pending"
- Update apply_queue:
  - if needs_confirmation_count > 0 => queue_status="needs_review"
  - else queue_status="ready"

I) Queue listing for UI
GET /api/agent/queue?resumeId=...
Return:
{
  "summary": { "total": N, "ready": X, "needsReview": Y, "prepping": Z, "failed": F },
  "items": [
    { "job": {...}, "fitScore": 0-100, "queueStatus": "...", "resumeVersionId": "...",
      "safeEditsAppliedCount": 5, "needsConfirmationCount": 2 }
  ]
}

J) Review payload
GET /api/agent/review?resumeVersionId=...
Return:
{ "resumeVersion": {...}, "suggestions": [...], "job": {...}, "scoreImpact": {...} }

K) Apply / reject / manual edit suggestions
POST /api/agent/apply-edits
Body:
{
  "resumeVersionId": "...",
  "acceptedEditIds": ["..."],
  "rejectedEditIds": ["..."],
  "manualEdits": { "editId": "edited text" }
}
Return updated: { content_json, suggestions, queueStatus }

Rule:
- When all NEEDS_CONFIRMATION are resolved (accepted/rejected), set queue_status="ready".

L) Export
POST /api/agent/export
Body: { resumeVersionId, format: "docx"|"pdf"|"both" }
Return: { taskId }

M) Apply clicked
POST /api/agent/apply-clicked
Body: { resumeId, jobId }
Return: { ok: true }
Client then opens apply_url in new tab.

N) Skip / retry
POST /api/agent/skip
Body: { resumeId, jobId }

POST /api/agent/retry
Body: { resumeId, jobId }
