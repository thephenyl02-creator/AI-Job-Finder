This will be used to:
- guide the LLM prompt
- populate “matchedTo” in the response

Do not over-engineer embeddings for V1 unless they already exist in the codebase.

6) LLM PROMPT (critical: truth-preserving, lawyer tone)
---------------------------------------
Create ONE LLM call for all bullets (batch) to reduce cost.

System instruction:
“You are a resume editor specializing in lawyers transitioning into legal technology roles. You must not invent experience, tools, outcomes, metrics, leadership, or responsibilities that are not implied by the original bullet. Your job is to translate and reframe the bullet using terminology that aligns with the job’s needs while staying truthful. Avoid buzzwords. Keep it concise.”

User prompt:
Provide:
- jobSignal (title/company/roleFamily/topSkills/topRequirements)
- bullets array
- for each bullet include its matched skill hints (top 1–3)

Ask model to return strict JSON:
{
  rewrites: [
    { original, suggested, why, matchedTo }
  ]
}

Rules for “suggested”:
- 1 sentence, max 28 words
- keep same factual meaning
- may add clarifying context ONLY if implied (e.g., “stakeholders” if bullet mentions clients/teams)
- do NOT add specific tools (Jira, SQL) unless present in the original bullet
- do NOT add metrics unless present in the original bullet

Rules for “why”:
- 1–2 sentences
- explicitly connect to jobSignal requirements (e.g., “This highlights contract lifecycle coordination and cross-functional execution.”)

If the bullet is too vague:
- keep suggested modest
- “why” should recommend what detail to add (e.g., “Add the type of agreements and the stakeholders you coordinated with.”)
